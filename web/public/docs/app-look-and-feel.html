Access, Login Security, and Override System Requirements

This document sets out the expected user roles, security controls, override workflow, and dashboard behaviour for the resource planning web app.

1. User Login & Authentication Requirements

1.1 User Roles

Role Purpose Typical Permissions
Administrator Full system control Create, edit, delete users; manage permissions; override bookings; view all data; reset locks.
Manager Team and project coordination Manage projects, allocations, and team-level booking actions; request or approve overrides; view relevant data.
Staff Day-to-day booking and allocation activity Book resources, view allocations, manage personal tasks, and follow role-based restrictions.
Viewer Read-only access View dashboards and booking information without editing permissions.
External / Client Limited outside access View shared information only, with restricted actions and no admin controls.

1.2 User Account Management

  • New accounts should be created by Admin only.
  • User accounts should be edited or deleted by Admin only.
  • Users should be able to update their own profile information, but not their title or permissions.

1.3 Password & Authentication Security

  • Password strength rules should be enforced.
  • Users should be able to reset their own password.
  • MFA is required and should remain in place.
  • Accounts should lock after repeated failed login attempts and require Admin reset.

1.4 Access Control

  • Pages and features should be restricted by role.
  • Staff should generally see their own activity while Admins can view all data.
  • Managers should see team-related data and may have limited cross-team visibility depending on policy.

2. Override System & Overbooking Prevention Requirements

2.1 Override Permissions

  • Only Administrators should be allowed to override booking conflicts.
  • Managers may be allowed to request an override, but it should require Admin approval unless an exception is approved.

2.2 Conflict Rules

  • The system should treat double-booking a resource as a conflict.
  • Some resources should be marked as protected or sensitive so they are highlighted and can be changed by Manager/Admin policy.

2.3 Override Workflow

  • Admin overrides should be applied instantly.
  • Manager overrides should require Admin approval.
  • Every override should require a reason, for example where a person is expected to be present all day but only needs the space for part of the day.

2.4 Logging & Notifications

  • All overrides should be logged with who performed them, when, and why.
  • Notifications should be available for overrides and review events.
  • Overrides should be reviewed periodically for accountability.

2.5 Unauthorized Override Attempt Alerts

  • Attempts to bypass booking rules without permission should be blocked.
  • Admins should receive alerts with username, time, resource, and attempted action.
  • In-app alerts and dashboard notifications should be supported.

3. Visibility & Dashboard Requirements

  • Different roles may see different dashboards, but the default experience should be shared allocation visibility.
  • Admins should see all data while staff should see their own relevant activity.
  • Daily, weekly, and monthly summary dashboards should be available.

4. Additional Security Considerations

  • Audit logs should record successful logins, failed login attempts, and logout events.
  • Inactive users should be logged out automatically after 30 minutes.
  • IP restrictions and location-based access controls should be considered as a good security enhancement.
